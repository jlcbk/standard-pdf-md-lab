# 标准 PDF 转 Markdown 调试工作计划

## 目标

为国家标准 PDF 建立一套可复现的 PDF 到 Markdown 转换流程。先用 `GB/T 13594-2025` 做基准样本，对比多个 GitHub 开源项目或本地工具的转换效果，再由人工审查输出结果，最终确定一个主方案和必要的后处理规则。

## 阶段 1：基线确认

1. 确认样本 PDF 信息：
   - 文件：`samples/GBT 13594-2025.pdf`
   - 页数、文件大小、MD5。
2. 抽样查看 PDF 内容结构：
   - 封面、目录、正文条款、表格、图、附录。
3. 建立评审模板：
   - 输出位置：`reviews/`
   - 记录候选工具、命令、耗时、输出文件、主要问题和评分。

## 阶段 2：候选项目筛选

优先测试 3 到 5 类方案：

1. 文档解析类：
   - 适合直接把 PDF 转成结构化 Markdown。
   - 重点看标题、列表、表格保留效果。
2. OCR / 版面分析类：
   - 适合扫描件或排版复杂 PDF。
   - 重点看版面分区、图表处理和中文识别。
3. PDF 文本抽取 + 后处理类：
   - 先抽文本，再用脚本恢复条款层级。
   - 重点看可控性和批处理稳定性。
4. LLM 辅助修复类：
   - 不作为第一步依赖，但可用于清理页眉页脚、修复表格或生成结构化目录。

每个候选方案都在 `candidates/<tool-name>/` 记录：

```text
source: GitHub URL 或工具来源
install: 安装命令
run: 转换命令
output: 输出文件路径
notes: 依赖、限制、失败原因
```

## 阶段 3：统一实验输出

每个工具使用同一个输入文件，输出到独立目录：

```text
outputs/<tool-name>/
  GBT-13594-2025.md
  assets/
  raw/
  run.log
```

要求：

- 保留原始命令输出日志。
- 不手工修改首轮输出，先保存工具原始效果。
- 如果需要二次清洗，另存为 `GBT-13594-2025.cleaned.md`。

## 阶段 4：人工评审

每个候选方案生成一份评审文件：

```text
reviews/<tool-name>.md
```

建议评分维度：

| 维度 | 分值 | 关注点 |
| --- | --- | --- |
| 标题层级 | 0-5 | 章节、条款、附录层级是否清晰 |
| 正文准确性 | 0-5 | 是否漏字、乱码、顺序错乱 |
| 表格保真 | 0-5 | 表格列、行、合并单元格是否可读 |
| 图像处理 | 0-5 | 图是否提取、引用是否保留 |
| 噪声清理 | 0-5 | 页眉页脚、水印、页码是否影响理解 |
| 自动化适配 | 0-5 | 是否容易批量处理、依赖是否稳定 |

## 阶段 5：确定方案

根据评审结果确定：

1. 主转换工具。
2. 是否需要 OCR 兜底。
3. 是否需要自定义后处理脚本。
4. Markdown 输出规范：
   - 标题格式。
   - 条款编号格式。
   - 表格格式。
   - 图片保存路径和引用方式。
   - 元数据字段。

最终沉淀为：

```text
scripts/convert_standard_pdf.sh
scripts/postprocess_standard_md.*
README.md 中的固定操作说明
```

## 下一步建议

1. 先联网筛选候选 GitHub 项目。
2. 为每个候选项目跑一次 `GB/T 13594-2025`。
3. 把首轮输出放入 `outputs/`。
4. 你审查 2 到 3 个最有希望的结果。
5. 根据审查意见做后处理脚本或换工具组合。
